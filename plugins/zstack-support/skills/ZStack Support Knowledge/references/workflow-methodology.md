# 渠道支持工作流方法论

工作流采用动态查证，而不是固定顺序全量查询。先判断入口，再按证据需要选择直答、追问、单点查证、多方并行查证或深查。默认工作模式为“支持工程优先”：具体 ZStack 支持事件必须主动查证，只有低风险概念问题才直接回答。

入口路由以 [查证路由规则](evidence-routing.md) 为准。

## 1. 入口判断

收到问题后先判断：

- 低风险直答：概念解释、通用机制、常见排查思路，直接回答。
- 支持事件默认查证：客户反馈、日志、告警、失败 API、错误文本、截图转写、版本异常、兼容性问题或客户回复口径，必须主动查证，但按问题类型路由来源。源码、机制、调用链、错误文本、类名、配置键、API、kvmagent、管理节点下发字段或代码定位必须先查 GitHub；历史相似案例查 BBS；已知缺陷、需求状态、影响版本和修复版本查 Jira；版本边界、标准口径和发布说明补充查 Confluence；OS、厂商和外部生态查 Tavily。
- 最小补充信息：缺少版本、错误文本、对象类型或操作路径，先问必要信息。
- 单点求证：只查用户指定来源。
- 多方并行查证：复杂故障、正式根因分析、升级/迁移/存储/网络/HA/数据风险。需要并行时请显式要求“用多 agent 并行深查”。
- 修复版本/回合确认：询问哪个版本修复、是否已经发布、是否合入指定版本线时，必须同时查 Jira/Confluence 与 GitHub commit/tag/branch。
- 深查：证据不足、来源冲突、调用链未追完、历史案例过多或用户明确要求。

具体 ZStack 支持事件不要求用户显式说“分析”也要主动查证。完整大报告只在用户明确要求完整分析、故障、根因、交接、闭环或提报研发材料时输出。

### 源码优先硬约束

凡是用户追问“源码呢”“代码里哪来的”“哪个字段下发”“调用链怎么走”“机制是什么”“这个错误是哪段代码抛的”，第一步必须查询 GitHub 源码、文件、commit、tag 或 release branch。Jira、Confluence、BBS、Tavily 只能作为补充来源，不能替代 GitHub 源码查证。

如果误先查了 Jira、Confluence、BBS 或 Tavily，必须立即切回 GitHub，并停止继续扩展非源码来源，直到 GitHub 查证完成或明确失败。

若 GitHub MCP 不可用，必须明确写“GitHub 源码查证未完成”，不得输出“源码已确认”“代码逻辑确认”“未合入某版本线”等源码或版本线结论。

## 1.1 多轮追问

如果用户追问“这个修复”“这个问题”“继续查”“有没有合到某版本”“那 Jira 呢”“源码呢”“有没有类似案例”，继承上一轮事件上下文继续查证，不从零开始。

Codex 技能触发是宿主行为，插件不能保证普通多轮追问自动重新进入技能。因此首次分析结束时提示用户后续追问继续使用 `@ZStackSupport:事件分析` 或 `ZStackSupport:事件分析`。如果用户已经重新指定技能，必须复用上一轮上下文。

## 4. 证据映射

对每个重要断言记录：

- 证据来源
- 时间戳或产物位置
- 公开标签：已确认 / 较可能 / 可能 / 证据缺失
- 证据级别：E0 / E1 / E2 / E3 / E4 / E5
- 边界：当前证据、源码参考、手册参考、ZStack知识社区(BBS) 参考、公开 Web/厂商论坛参考或 Jira/Confluence 内部参考

证据级别定义：

```text
E0：无证据，纯假设
E1：仅客户陈述或截图
E2：部分日志、命令输出或孤立产物
E3：仅有 ZStack知识社区(BBS)/公开 Web/厂商论坛/历史参考
E4：完整当前证据链或可复现路径
E5：修复、发版、复测或客户验证闭环
```

规则：

- E0-E2 不能作为最终根因
- E3 不能单独关闭当前事件
- 主诊断通常需要 E4，否则保持为排序假设
- 闭环通常需要 E5，否则闭环决策必须明确说明未闭环部分

## 5. 按需参考来源

公开源码、手册、BBS、外部 Web、Jira 和 Confluence 用于解释机制、提供历史、生态或内部边界参考，不能证明客户环境实际行为。

| 来源 | 适用问题 | 边界 |
|------|----------|------|
| GitHub 源码 | API、类名、配置键、错误文本、调用链、源码机制、版本差异 | 公开源码参考，非当前客户证据 |
| 官网文档 | 配置含义、官方限制、操作步骤、产品功能边界 | 官方文档参考，非当前客户环境证据 |
| ZStack知识社区(BBS) | 历史相似案例、相似症状、可复用验证动作 | 历史参考，不能单独关闭事件 |
| Tavily 外部 Web | OS、内核、QEMU/KVM、libvirt、Ceph、GPU、AI 框架等外部生态 | E3 公开参考，必须使用脱敏查询词 |
| Jira | 已知缺陷、需求编号、修复状态、影响版本、修复版本、组件归属、研发结论 | E3 内部参考，只读查询；内部同事协作可输出编号、标题摘要和可点击链接，不能输出原文、评论或附件 |
| Confluence | 内部说明、版本边界、操作规范、发布说明补充、兼容性矩阵、产品口径 | E3 内部参考，只读查询；内部同事协作可输出标题摘要和可点击链接，不能输出原文或附件 |

### 默认并发查证

普通事件分析默认由主 agent 查证汇总。只有用户显式要求“多 agent / 子 agent / subagent / 并行深查 / 多路并行深查”时，才尝试发现并使用当前 Codex 宿主暴露的 subagent 调度工具并发查证，例如 `multi_agent_v1.spawn_agent`。若宿主未暴露、不允许或调用失败，必须明确标注未触发原因并降级为主 agent 查证；不得假装已经并发。

推荐拆分：

```text
源码/版本 agent：只查 GitHub 源码、commit、tag、release branch、调用链、版本差异；不得查询 Jira、Confluence、BBS 或 Tavily。
历史案例 agent：ZStack知识社区(BBS) 相似案例、差异、可复用验证动作。
内部跟踪 agent：Jira 缺陷/需求状态、影响版本、修复版本、关联项；必要时补 Confluence 内部口径。
文档/外部 agent：官网文档、Confluence 文档边界、Tavily 厂商/OS/外部生态资料。
```

主 agent 保持总控，负责当前证据整理、问题边界、最终合并和结论约束。若当前会话未暴露 subagent 工具，必须标注“Subagent 状态：未触发”，并由主 agent 继续查证。

### Jira / Confluence 调用规则

Jira 优先用于确认缺陷和需求状态。查询时优先使用明确编号，其次组合错误文本、组件名、版本号、OS、存储、网络、升级等关键词。输出保留工单号、Markdown 链接、标题摘要、状态、影响版本、修复版本、组件和可公开行动建议。Jira/TIC/SUG/BUG 链接必须是完整 URL，格式为 `http://jira.zstack.io/browse/<KEY>`。

Confluence 优先用于确认内部文档、版本边界、操作规范和发布说明补充。查询时组合产品名、功能名、版本号、错误文本、OS 或第三方组件关键词。输出保留文档标题摘要、Markdown 链接、适用版本、版本边界和可公开行动建议。

Jira 与 Confluence 可以同时查询，但只在正式根因分析、兼容性/升级风险、客户口径或来源冲突时使用。若内部来源与公开源码、BBS 或官网文档冲突，标注“来源冲突”，保持保守结论并说明需要进一步确认。

Jira/Confluence 必须通过插件声明的只读共享远端 `zstack_atlassian_shared` MCP 查询。不要使用旧的 `zstack_atlassian` 本机适配器、`support_archive`、`support_sql_analyst` 或其他旧全局 MCP 目标替代 Jira/Confluence；这些旧目标不属于本插件事件分析证据来源。

### 修复版本 / 回合确认规则

当用户问“哪个版本修复”“是否已经发布”“是否合入 4.8.x/5.x/某版本线”“fixVersion 是否可信”时，不允许只靠 Jira fixVersion、BBS 历史案例或 Confluence 口径下最终结论。

必须同时查：

- Jira：内部缺陷/需求状态、fixVersion、影响版本、关联 issue。
- GitHub：commit、commit message、PR/merge 线索、tag、release branch、相关文件差异。
- Confluence/发布说明：仅在版本边界不清时补充。

结论边界：

- 只有 Jira fixVersion 时，只能说“内部跟踪标记/规划为某版本”。
- 要确认“已修复”，必须有提交、发布版本或明确发布说明证据。
- 要确认“未合入某版本线”，必须有目标 tag/branch 查证、提交不可达、发布说明排除或明确未回合记录。
- GitHub 查询未完成时，必须明确“不能对是否合入指定版本线下最终结论”。

统一证据块：

```text
来源：
查询词：
命中内容：
相关性：高 / 中 / 低
能支持的判断：
不能支持的判断：
证据边界：
下一步是否需要深查：是 / 否
```

来源状态使用：

```text
工具可见 / 已连接 / 结构化查询成功 / MCP 查询未完成 / 未配置
```

## 9. 交接

交接时包含问题摘要、影响范围、时间线、关键对象、最强证据、已确认、较可能、可能、证据缺失、已执行操作、已排除方向、参考证据边界、下一责任人、客户安全下一步行动。
